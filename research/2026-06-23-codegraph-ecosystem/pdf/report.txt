1 结论先行
1.1 一句话结论
OmniWeave 的核心优势不在“支持更多语言”、不在“有一个图数据库”、也不在“比 grep 更聪明”。真
正的优势是：
它把 coding agent 最容易付出高成本、且 LSP/grep 结构性看不见的跨边界关系，做成
了可追踪、可置信、可预算控制的本地图。
这类项目的生态正在分成三条路：第一条是 结构图/代码智能后端，代表是 codegraph、CodebaseMemory、CGC、Codanna、code-review-graph；第二条是 语义记忆/向量检索，代表是 Claude Context、
semantic-search-mcp、Cursor 类索引；第三条是 工业级精确代码索引/安全分析，代表是 SCIP/Sourcegraph、Kythe、Glean、CodeQL、Joern、Semgrep/OpenGrep。OmniWeave 应该站在第一条，但吸
收第二条作为入口、第三条作为可选事实来源，而不能被任何一条吞掉。

1.2 OmniWeave 当前最强的五个点
1. 边界关系是第一公民。 OmniWeave 明确抓 Python/JS/TS/Go/workflow 到本地脚本、R S4 dispatch、Snakemake/Nextflow artifact DAG、external tool invocation。这不是“多语言列表”，而是 LSP
和 grep 的盲区建模。README 将这一点写得非常清楚：cross-language、cross-process、dynamic
dispatch、workflow DAG、external-tool graph 是产品主线，见 README.md:22-31,135-164。
2. 诚实边界强。 generalCrossLangEdges 对动态路径、插值 basename、非真实 subprocess
调用做 gate；推断边带 provenance、metadata.confidence、registeredAt，静态不
可解就跳过，见 src/resolution/callback-synthesizer.ts:1941-2039。这比“什
么都连上”的图更适合 agent，因为错边比漏边更危险。
3. 价值经过 agent A/B 约束。 OmniWeave 的 README 和 eval 资料没有把工具包装成“更正确”，
而是承认正确性常常与 grep/LSP 打平，优势主要在 reverse/blast-radius 的工具调用、token、延
迟和稳定性，见 README.md:35-98 和 eval-results/agent-ab-2026-06-13/。
4. 架构复杂度克制。当前主链是 WASM tree-sitter extraction → SQLite/FTS → resolution/synthesizer
→ MCP。没有默认外部向量库、没有多后端图数据库、没有把 LLM 文档生成塞进核心。
CHECKPOINT 的模块边界和 skip-over-guess 原则与此一致，见 CHECKPOINT.md:38-55,
77-88。
5. agent 输出面被当作产品，而不是 API 附属物。继承自 codegraph 的输出预算、explore/node
作为 primary tool、工具 surface 收缩等思想，使图不是“给人查的数据库”，而是“给 agent 消费的
行动上下文”。codegraph 的 src/mcp/tools.ts 已经把默认工具压到 explore，OmniWeave
README 当前默认五个核心工具，方向一致。

1.3 最大的战略边界
OmniWeave 不应试图成为所有东西：
• 不应替代 LSP 在同语言精确导航上的优势。LSP/SCIP 有编译器语义，OmniWeave 的价值在跨
边界和零配置。
• 不应用向量检索伪装成结构事实。语义搜索适合“auth 在哪儿”这种概念入口，但不能直接产
生 calls/crossLang/overrides 这类可置信边。
• 不应追求 Codebase-Memory/CGC 式大工具面。14 个、20 个、甚至更多 MCP tools 会提高试错
概率；OmniWeave 更应该把少数工具做到足够完整。
• 不应为了“企业感”引入多图数据库、多 UI、LLM 文档生成、自然语言 Cypher 等核心依赖。那
些是外围产品层，不是最小可信 substrate。

2.3 反营销处理原则
这类项目普遍喜欢宣称“百万行秒级索引”“数百倍 token 节省”“支持上百语言”。本报告采用三条过
滤规则：
1. 项目自述数字只标注来源，不直接变成结论。例如 Codebase-Memory 的 Linux kernel 级索
引数字、Codebase-Memory paper 的 83% answer quality、code-review-graph 的 82x/528x token
reduction，都按来源分级。
2. 语言数量不是主要指标。关键是该语言是否有可用关系、是否能跨文件/跨进程、是否有
false-positive gate、是否能被 agent 直接消费。
3. 工具数量不是优势。 MCP 工具越多，agent 选择成本越高。真正优势是少数工具是否能一次
返回可行动上下文。

3 生态分层：这类项目到底在竞争什么
3.1 四个真正的问题
所有“code graph / codebase memory / code intelligence for agents”项目，本质上都在回答四个问题：
1. 如何把代码变成可查询的事实？选择 Tree-sitter、LSP、SCIP、compiler indexer、regex、embedding，
决定了事实的可信度。
2. 如何把事实保持新鲜？选择 file watcher、git diff、hash/Merkle、incremental rebuild、snapshot
artifact，决定了开发时可用性。

3. 如何把事实交给 agent？选择 MCP tools、skills、slash commands、CLI、IDE extension、query
language，决定了 agent 会不会用错。
4. 如何证明它值得存在？选择 unit eval、agent A/B、retrieval benchmark、security finding、token
accounting，决定了项目是否会陷入自我宣传。

4.12 SCIP/Sourcegraph：精确索引的事实格式，不是查询引擎本身
4.12.1 定位

• 设计文档反对直接编码全图 adjacency list，理由是并行性、内存和 consumer 侧负担，见
repos/scip/docs/DESIGN.md:97-123。

4.12.3 OmniWeave 可借鉴
1. SCIP importer，而非 SCIP core。OmniWeave 可读取 index.scip，导入 precise same-language
references/inheritance，标注 provenance=scip。
2. Symbol ID 规范。 OmniWeave 当前 ID 偏本地 DB；未来 snapshot/bundle/export 可参考 SCIP
的 package-aware symbol scheme。
3. Streaming artifact。如果做团队共享索引，不要直接 dump 全图 JSON；应支持 streaming、
version、root path、position encoding。

4.13 Kythe/Glean/CodeQL/Joern/Semgrep/OpenGrep：重型工业系统的启示
这些系统没有全部本地 clone 深读，本报告基于官方文档/公开资料定位：
• Kythe：强调跨语言 semantic code graph、facts/edges、serving tables、cross-reference。适合大规
模公司代码导航，不适合轻量 agent MCP core。
• Glean：Meta 开源的代码索引系统，使用 Angle query language，目标是多语言代码索引与查
询。它代表“Datalog-like facts + query”的工业路线。
• CodeQL：GitHub 的 semantic code analysis engine，把代码作为数据库查询，主要用于 variant
analysis/security。强在安全分析，不在低延迟 agent 上下文。
• Joern：Code Property Graph，AST/CFG/PDG 等统一为 CPG，用 Scala DSL 挖漏洞模式。适合
安全研究，不适合作为日常 agent exploration 默认层。
• Semgrep/OpenGrep：规则/模式匹配、SAST、taint/reachability。适合“找安全/规范模式”，不是
repo understanding graph。
结论：OmniWeave 应借鉴它们的事实模型纪律、query safety、provenance、版本化 schema、
可解释规则，而不是照搬重型查询语言或安全扫描平台。

4.14 RepoGraph 与 CodeRAG-Bench：研究评测提醒我们别只测检索
RepoGraph 把 repo-level code graph 加入 SWE-bench/Agentless/SWE-agent。源码显示它大量借
鉴 Aider repo-map、Agentless、grep-ast；construct\_graph.py 中 甚 至 有 对 Python 代 码 做
exec(import_statement) 来识别标准库函数的逻辑，风险很高，不适合作为生产参考。
CodeRAG-Bench 关注“检索增强是否提升代码生成”。它的 retrieval 脚本覆盖 BM25、SBERT、
OpenAI/Voyage embeddings、repoeval、swe-bench repo retrieval，generation 侧再评估模型输出。这
对 OmniWeave 的启发不是实现，而是评测：只测 Recall@K 或 token saving 不够，还要测最终 agent
task 是否更快、更稳、更少误读。

• SQLite 是 OmniWeave 当前最优核心。 codegraph、OmniWeave、code-review-graph、CodebaseMemory、sdsrss/code-graph-mcp 都选择 SQLite 或 SQLite 衍生能力。它足够本地、可携带、低

5.3 增量同步
生态里主要有五种：
1. 文件 watcher + debounce：codegraph/OmniWeave、semantic-search-mcp；适合开发态，但要处
理平台 watcher 差异和 stale state。
2. Git diff/status：Codebase-Memory、code-review-graph；适合 review/CI，但对未跟踪文件不完
整。
3. Merkle/hash tree：sdsrss/code-graph-mcp；no-change detection 快，适合大 repo。
4. snapshot artifact：Codebase-Memory；适合团队共享和冷启动。
5. remote/cloud collection sync：Claude Context；状态复杂度最高，必须有 lock/recovery。
OmniWeave 当前 watcher + staleness banner 是合理核心；下一步最值得加的是 hash-verified
snapshot import/export，不是云同步。

6 OmniWeave 的优势：不是“大而全”，而是边界可信
6.1 优势一：抓住 LSP/grep 共同盲区
grep 的盲区是“关系”；LSP 的盲区是“跨语言/跨进程/工作流/动态框架约定”。大部分竞品只解决其
中之一：
• Aider 解决“哪些文件值得读”，不是“谁调用谁”。
• Serena/LSP 解决同语言 symbol navigation，不解决 Python→R、Snakemake→tool。
• Claude Context 解决“概念上相关”，不解决 structure。
• CodeQL/Joern 解决安全程序分析，不解决 agent 日常上下文形态。
OmniWeave 的 crossLang、S4 dispatch、workflow DAG、external tool nodes 正好打在这些交
集盲区。这个定位应坚持。

6.2 优势二：诚实边界比高召回更稀缺
很多项目把“能猜到”当成“能连边”。OmniWeave 当前更克制：
• 动态路径里 residual \{...\}/\$ 就 skip；
• interpreter 必须处在真实 subprocess call 里，不能只是 echo 字符串；
• fan-out 有 cap；
• heuristic edge 带 confidence；
• static dispatch 只建声明骨架，不伪造运行时 target。
这正是 agent substrate 的信任基础。Agent 会把结构边当成事实来行动；错边会导致错误修改、
错误影响面、错误审查结论。OmniWeave 的“宁可漏，不可骗”是优势，不是保守。

6.3 优势三：A/B 证明了真实价值边界
README 当前最有力量的不是数字本身，而是承认边界：correctness 经常 tie；single-point lookup
不赢；concept search 不是 OmniWeave；cross-process at scale 由于运行时路径构造也会 tie。很多竞
品还停在“我减少了多少 token”的自述阶段，OmniWeave 已经进入“在哪些 query shape 赢、在哪些
不赢”的阶段。
这会带来一个战略好处：产品路线可以少走弯路。比如：
• 不应该把资源投向“让 symbol lookup 比 grep 更快很多”；
• 应该投向 reverse/blast-radius、workflow/crossLang、weak-model guardrail、output sufficiency；
• 不应该把 semantic search 写成核心卖点；
• 可以把 semantic search 作为 seed layer 服务 explore。

6.4 优势四：本地、轻量、可分发
Codebase-Memory、Claude Context、CGC、Blarify 都有更重的状态面：外部 vector DB、多 DB
backend、SCIP/Docker/LSP、cloud collection、Neo4j/Falkor。OmniWeave 的核心依赖更少：Node
22.5+、WASM tree-sitter、SQLite。这是安装、CI、隐私、团队采用的实际优势。
但要注意：轻量不是简陋。轻量必须和 artifact、version、integrity、watch freshness、schema
migration 一起做，否则只是在本地临时可用。

7.4 优先级 P2：semantic sidecar
借鉴 Claude Context、semantic-search-mcp、Cursor 的方向，但不做默认依赖：
• semantic sidecar 只返回 candidate file/symbol IDs；

7.5 优先级 P2：review vertical skill
借鉴 code-review-graph 的场景化：
• /omniweave:review-delta：自动取 git diff changed files → impact → affected workflows/tools → test gaps；
• /omniweave:review-pr：base branch diff → risk summary；
• 输出必须短、稳定、可粘贴，不暴露整个 graph JSON；
• 所有 review 指标都要标明 graph-derived 上界还是 git co-change evidence。

8 OmniWeave 不应该借鉴什么
8.1 不要默认追求大工具面
Codebase-Memory 的 14 tools、CGC 的大工具面、code-review-graph 的 vertical tools 都各有道理，
但 OmniWeave 的默认 MCP surface 必须克制。对 agent 来说，多一个工具不是多一个能力，而是
多一个误选路径。
判断标准：
如果一个新增 tool 不能在 A/B 中降低工具调用、降低 token、减少 fallback read、或提
升稳定性，就不应默认暴露。

8.2 不要引入多数据库后端
Neo4j、FalkorDB、Kuzu、Ladybug、remote graph DB 都可以做 enterprise story，但会破坏 OmniWeave
的 local-first 和零配置优势。SQLite 已足够支持当前核心；更值得做的是 schema、index、query
plan、snapshot，而不是 backend abstraction。

8.3 不要把 LLM 文档生成塞进核心
Blarify、Graphify、Understand Anything 都做 documentation/report/tour。这些适合 offline product layer，
不适合 indexer core。LLM 生成内容不是事实，必须和 graph facts 分开存储、分开 provenance。

8.4 不要把语义搜索当结构事实
向量检索可以帮助发现入口，但它不能回答调用关系、影响半径、workflow artifact、S4 dispatch。
语义层若进入核心边表，会破坏 graph trust。

8.5 不要复制安全分析平台
CodeQL/Joern/Semgrep/OpenGrep 很强，但它们解决的是 variant analysis、CPG mining、taint/rule
matching。OmniWeave 可在未来以 plugin 接入安全 pass，但不应把 query DSL 或 taint engine 变成
MCP 默认能力。

9 建议路线图
9.1 Phase A：输出与工具面硬化
1. 为 explore/node/callers/impact 建 query-shape goldens：single-point、reverse、flow survey、
file read replacement、crossLang、workflow。
2. 每个工具输出记录：字符数、source snippets 完整率、omitted reason、fallback read rate。
3. A/B 默认工具面：5 tools vs 3 tools vs explore-only，指标为工具调用、token、turns、正确率、模
型误选率。
4. 引入 continuation key：一个 symbol/file/flow 被返回后，后续工具可用 key 直接定位。
Done when：默认工具面比当前不增加 correctness 风险，同时降低平均 tool mis-pick 或 fallback
read。

9.2 Phase B：snapshot artifact
1. 定义 omniweave-snapshot.json manifest。
2. 支持 export/import read-only snapshot。
3. 引入 schema version and compatibility check。
4. 做 corrupt snapshot、wrong root、stale source、malicious path regression tests。
Done when：干净 checkout 可在无全量 reindex 的情况下读 snapshot，但任何源码变更都能触
发 stale warning/reindex path。

9.3 Phase C：SCIP importer
1. 读取 SCIP protobuf，导入 definitions/references。
2. 映射到 OmniWeave nodes/edges，保留 original symbol string。
3. 输出中展示 provenance=scip。
4. 与 Tree-sitter edge 冲突时不静默覆盖。
Done when：在 TS/Java/Rust fixture 上 same-language references 明显更精确，且不影响 crossLang/workflow/S4 gates。

9.4 Phase D：semantic sidecar
1. 先做 internal seed，不暴露为默认 MCP tool。
2. 本地 provider 优先，外部 provider 明确 opt-in。
3. Eval 只测“是否帮助 concept query 找到正确 explore seed”，不把它混入 structural correctness。
Done when：concept query 的 first useful file rate 提升，且 structure query 无 regression。

9.5 Phase E：review vertical skill
1. 用现有 impact/workflow/tool edges 生成 PR review context。
2. 输出 risk/test gaps/affected tools/affected workflows。
3. 区分 graph-derived upper bound 和 git co-change evidence。
4. 不默认安装，作为 skill/plugin 分发。

10 最终判断
OmniWeave 应该避免两个诱惑：一个是向上变成“通用知识图/可视化平台”，一个是向下变成“更
快 grep/更多语言 parser”。前者会拖垮核心，后者会失去差异化。
最强路径是：
把 OmniWeave 做成 coding agent 的可信结构控制层：少工具、强输出、边界可信、跨
语言/跨进程/workflow/dynamic-dispatch 优先、本地可分发，并用 A/B 证明每个新增
能力真的减少 agent 成本。
这也是与所有竞品拉开距离的地方。Codebase-Memory 更像“代码库记忆平台”；Serena 更
像“IDE/LSP agent 工具层”；Claude Context 更像“语义召回层”；CodeQL/Joern 是“安全分析层”；
Graphify/Understand Anything 是“学习/可视化层”。OmniWeave 的最佳位置不是吞并它们，而是成
为它们之下或旁边的可信结构骨架。
